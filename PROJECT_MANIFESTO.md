# Task Tracker — Project Manifesto

Этот документ — единая точка правды о проекте. Его читает и ментор (Claude), и автор (я). Он обновляется по мере развития проекта и всегда отражает текущее состояние.

## Цель проекта

Task Tracker — бэкенд-приложение для управления личными задачами с амбицией вырасти в многопользовательский органайзер с социальной и игровой составляющей.

Параллельно проект выполняет **образовательную функцию**: автор использует его для глубокого изучения Kotlin, бэкенд-разработки на Ktor, и принципов Domain-Driven Design.

## Контекст автора

- Бэкграунд: Python (Django, FastAPI, SQLAlchemy) на учебном уровне, без продакшен-опыта.
- Знаком с базовыми концепциями hexagonal architecture и DDD (см. существующий проект `hospi_api`).
- Новый для языка: Kotlin.
- Новый для экосистемы: Ktor, Exposed, Gradle, JVM.
- Среда разработки: IntelliJ IDEA Ultimate 2025.3.4, JDK 21, macOS.

## Философия работы

1. **Автор пишет код сам.** Ментор объясняет, направляет, ревьюит, но не пишет за автора.
2. **YAGNI.** Не делаем того, что не нужно сейчас. Амбиции идут в roadmap, не в текущий код.
3. **Bounded contexts.** Каждая крупная область — отдельный модуль. Расширение = добавление контекста, а не раздувание существующего.
4. **Ubiquitous Language.** Одно слово — один смысл в рамках контекста. Никаких универсальных «задач, которые умеют всё».
5. **Осознанность решений.** Каждый архитектурный выбор проговаривается, trade-off'ы называются явно.

## Архитектурные принципы

Применяются с v1. Не как заделы на будущее, а как правильная форма кода.

- **Hexagonal architecture (ports & adapters)**: домен не знает про БД, HTTP, сериализацию.
- **Модульная структура Gradle**: отдельные модули `domain`, `application`, `infrastructure`, `api`, `app`.
- **Типизированные идентификаторы**: `UserId`, `TaskId` как value classes, не голые примитивы.
- **UUID вместо автоинкремента**: генерация ID на стороне приложения.
- **Domain events с первого дня**: даже если некому подписываться, места для подписки существуют.
- **Dual-model persistence**: domain-модель и persistence-модель отдельно, между ними маппинг.
- **Sealed hierarchies** для моделирования конечных множеств состояний и вариантов.

## Scope v1 (минимум, рабочий прототип)

Только личный to-do tracker для одного пользователя. Срок: ~2 недели по вечерам.

**Фичи:**
- Task с полями: id (UUID), text, due_date (опционально), status.
- Статусы: NEW, IN_PROGRESS, DONE, CANCELED.
- Операции: создать задачу, получить список, получить по id, начать (start), завершить (complete), отменить (cancel), вернуть в работу (reopen), отредактировать текст, изменить/задать/убрать срок, удалить.
- Хранение: PostgreSQL через Exposed.
- Транспорт: REST API на Ktor с JSON (kotlinx.serialization).
- Тесты: unit (домен), integration (репозиторий), e2e (API).
- Запуск: Docker Compose для БД.
- Пользователь: один, захардкожен, без авторизации.

**Чего нет в v1:**
- Подзадач и вложенности
- Регулярных задач
- Комментариев
- Заметок
- Групп и мультипользовательности
- Делегирования
- Геймификации
- Telegram-бота

## Roadmap

- **v2** — комментарии к задачам + подзадачи (вложенность).
- **v3** — регулярные задачи (шаблоны + планировщик). Сюда же позже приклеится автопополнение.
- **v4** — многопользовательскость и аутентификация (отдельный bounded context Users).
- **v5** — группы и совместные задачи.
- **v6+** — заметки, делегирование задач, покупки, геймификация, Telegram-бот. Каждый пункт — отдельный bounded context.

Сроки после v1 не заданы и подвижны. Roadmap может меняться по мере получения опыта.

## История решений

(Сюда пишутся важные решения с обоснованием. Формат: дата — решение — почему.)

- **2026-04-21** — Язык проекта: Kotlin. Почему: выразительность, null-safety, общая экосистема с Android (будущий клиент).
- **2026-04-21** — Стек: Ktor + Exposed (не Spring). Почему: меньше магии, лучше для обучения, Kotlin-native.
- **2026-04-21** — Scope v1 ограничен одним пользователем без авторизации. Почему: минимизировать количество одновременно изучаемых технологий.
- **2026-04-21** — ID через UUID, не автоинкремент. Почему: независимость от БД, совместимость с будущим распределением по контекстам.
- **2026-04-21** — Доменная модель Task: named commands вместо общей операции смены статуса. Почему: соответствие принципу ubiquitous language, естественная публикация domain events, локальность инвариантов, соответствие DDD-подходу «моделируем поведение, а не данные».
- **2026-04-21** — Добавлен статус CANCELED. Почему: закрывает сценарий «задача не будет выполнена» без потери истории и без необходимости врать системе через DONE.
- **2026-04-21** — Матрица переходов: из NEW — в IN_PROGRESS, DONE, CANCELED. Из IN_PROGRESS — в DONE, CANCELED. Из DONE — только обратно в IN_PROGRESS через reopen. Из CANCELED — никуда. Почему: CANCELED и DONE как (преимущественно) терминальные состояния сохраняют семантику статусов; единственное исключение — reopen из DONE, для случаев ошибочного завершения.
- **2026-04-21** — Редактирование текста разрешено в любом статусе, включая терминальные. Обоснование: опечатку в формулировке неприятно видеть в истории.
- **2026-04-21** — Изменение/удаление/задание due_date разрешено только в нетерминальных статусах (NEW, IN_PROGRESS). Обоснование: срок имеет смысл только у активной задачи, у завершённой или отменённой задачи он зафиксирован как часть истории.
- **2026-04-22** — События TaskTextChanged и TaskRescheduled не публикуются. Почему: правка текста и срока — коррекция ошибки ввода, не часть жизненного цикла задачи. Подписчиков сейчас нет и в roadmap не видно. По YAGNI — не добавляем до появления реальной потребности.
- **2026-04-22** — Инвариант поля text модели Task: не пустой, длина ≤ 1000 символов. Почему: иначе это не задача.
- **2026-04-22** — Инвариант dueDate — не может быть в прошлом. Почему: срок «вчера» при создании противоречит смыслу дедлайна.
- **2026-04-22** — Структура сборки: multi-module Gradle с самого начала (не single-module с пакетами). Почему: compile-time enforcement архитектурных границ (domain не может импортировать инфраструктуру даже случайно), явное выражение направления зависимостей в build-файлах, Gradle как стандарт JVM-индустрии — полезно освоить.
- **2026-04-22** — Интерфейсы без суффикса, реализации с префиксом технологии. Почему: стандарт индустрии.
- **2026-04-22** — Пять модулей, а не четыре: domain, application, infrastructure, api, app. Почему: изоляция composition root, api не должен знать про infrastructure.
- **2026-04-22** — Версии — через libs.versions.toml, общие настройки — через convention plugins в build-logic/. Почему: single source of truth для версий; не дублировать настройки между модулями; стандарт современного Gradle.
- **2026-04-22** — Дата в domain — java.time.LocalDate. Почему: JVM-only для бэкенда оправдан, Android-клиент будет отдельным проектом и общается по HTTP; YAGNI по отношению к multiplatform.
- **2026-04-22** — Версии v1: Kotlin 2.3.20, Ktor 3.4.1, Exposed 1.0, PostgreSQL 17, JDK 21
- **2026-04-22** — Тестовый фреймворк — Kotest. Почему: Kotlin-идиоматичный, выразительные матчеры, встроенный property-based testing. Отвечает духу проекта (учить Kotlin во всех слоях).
- **2026-04-22** — Kotest: FunSpec как дефолт, BehaviorSpec — для e2e и многошаговых сценариев. Почему: вложенность given/when/then избыточна для простых утверждений; BDD-стиль оправдан только там, где есть реальный сценарий с контекстом и шагами. Почему: последние стабильные на момент старта; Exposed 1.0 — первый stable API
