# Task Tracker — Project Manifesto

Этот документ — единая точка правды о проекте. Его читает и ментор (Claude), и автор (я). Он обновляется по мере развития проекта и всегда отражает текущее состояние.

## Цель проекта

Task Tracker — бэкенд-приложение для управления личными задачами с амбицией вырасти в многопользовательский органайзер с социальной и игровой составляющей.

Параллельно проект выполняет **образовательную функцию**: автор использует его для глубокого изучения Kotlin, бэкенд-разработки на Ktor, и принципов Domain-Driven Design.

## Контекст автора

- Бэкграунд: Python (Django, FastAPI, SQLAlchemy) на учебном уровне, без продакшен-опыта.
- Знаком с базовыми концепциями hexagonal architecture и DDD (см. существующий проект `hospi_api`).
- Новый для языка: Kotlin.
- Новый для экосистемы: Ktor, Exposed, Gradle, JVM.
- Среда разработки: IntelliJ IDEA Ultimate 2025.3.4, JDK 21, macOS.

## Философия работы

1. **Автор пишет код сам.** Ментор объясняет, направляет, ревьюит, но не пишет за автора.
2. **YAGNI.** Не делаем того, что не нужно сейчас. Амбиции идут в roadmap, не в текущий код.
3. **Bounded contexts.** Каждая крупная область — отдельный модуль. Расширение = добавление контекста, а не раздувание существующего.
4. **Ubiquitous Language.** Одно слово — один смысл в рамках контекста. Никаких универсальных «задач, которые умеют всё».
5. **Осознанность решений.** Каждый архитектурный выбор проговаривается, trade-off'ы называются явно.

## Архитектурные принципы

Применяются с v1. Не как заделы на будущее, а как правильная форма кода.

- **Hexagonal architecture (ports & adapters)**: домен не знает про БД, HTTP, сериализацию.
- **Модульная структура Gradle**: отдельные модули `domain`, `application`, `infrastructure`, `api`.
- **Типизированные идентификаторы**: `UserId`, `TaskId` как value classes, не голые примитивы.
- **UUID вместо автоинкремента**: генерация ID на стороне приложения.
- **Domain events с первого дня**: даже если некому подписываться, места для подписки существуют.
- **Dual-model persistence**: domain-модель и persistence-модель отдельно, между ними маппинг.
- **Sealed hierarchies** для моделирования конечных множеств состояний и вариантов.

## Scope v1 (минимум, рабочий прототип)

Только личный to-do tracker для одного пользователя. Срок: ~2 недели по вечерам.

**Фичи:**
- Task с полями: id (UUID), text, due_date (опционально), status.
- Статусы: NEW, IN_PROGRESS, DONE.
- Операции: создать, список, получить по id, изменить статус, удалить.
- Хранение: PostgreSQL через Exposed.
- Транспорт: REST API на Ktor с JSON (kotlinx.serialization).
- Тесты: unit (домен), integration (репозиторий), e2e (API).
- Запуск: Docker Compose для БД.
- Пользователь: один, захардкожен, без авторизации.

**Чего нет в v1:**
- Подзадач и вложенности
- Регулярных задач
- Комментариев
- Заметок
- Групп и мультипользовательности
- Делегирования
- Геймификации
- Telegram-бота

## Roadmap

- **v2** — комментарии к задачам + подзадачи (вложенность).
- **v3** — регулярные задачи (шаблоны + планировщик). Сюда же позже приклеится автопополнение.
- **v4** — многопользовательскость и аутентификация (отдельный bounded context Users).
- **v5** — группы и совместные задачи.
- **v6+** — заметки, делегирование задач, покупки, геймификация, Telegram-бот. Каждый пункт — отдельный bounded context.

Сроки после v1 не заданы и подвижны. Roadmap может меняться по мере получения опыта.
